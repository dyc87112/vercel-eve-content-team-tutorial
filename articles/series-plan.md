# Vercel Eve 实战系列策划

## 一句话定位

围绕一个真实需求，持续开发一套 **SpringForAll AI 内容运营团队**：从第一位能在网页中对话的内容负责人开始，逐步加入本地资料处理、内容方法、长任务恢复、多 Agent 协作、人工审批、运营界面和 Vercel 部署。

系列不是逐项介绍 Eve API，而是回答一个贯穿始终的问题：

> 一个只会分析需求的 Agent，怎样一步步成长为可操作、可审查、可部署的内容运营系统？

## 为什么选择内容运营场景

SpringForAll 面向 Java 和 Spring 开发者，内容生产天然包含策划、研究、写作、审校和发布等不同岗位，适合展示 Eve 的目录式 Agent 组织方式。

这个案例还有几个重要优势：

- 不要求读者准备 Spring Boot 项目或其他业务系统；
- 基础资料和任务素材都可以随示例项目提供；
- 每个阶段都能直接产出 Markdown，结果容易观察和验证；
- 可以自然引出长任务、资料隔离、子 Agent、审批和评测；
- 最终可以形成一个真正可操作的内容运营台，而不只是命令行工作流。

## 最终交付物

系列完成后，用户可以在网页中创建内容任务，并完成以下流程：

1. 内容负责人分析需求并判断资料是否充分；
2. 选题策划给出多个方向，等待人工确认；
3. Java 技术研究员根据提供的资料建立事实底稿；
4. 技术作者根据已确认的大纲撰写文章；
5. 技术审校员检查事实、版本、代码与文章一致性；
6. 未通过的内容退回对应岗位修改；
7. 发布编辑生成正文、摘要、标题备选和发布检查清单；
8. 用户在运营台查看进度、批准或退回成果，并下载最终内容包；
9. 系统部署到 Vercel，并具备基本的观测、评测和回滚方案。

最终系统默认不依赖公众号、GitHub、数据库或其他业务平台。账号资料、历史文章和示例素材随项目提供，模型凭据是唯一必要的外部配置。

## 最终团队结构

```text
用户 / 内容运营台
        │
        ▼
内容负责人 Coordinator
        │
        ├── 选题策划 Topic Planner
        ├── Java 技术研究员 Researcher
        ├── 技术作者 Writer
        ├── 技术审校员 Reviewer
        └── 发布编辑 Publisher
        │
        ▼
本地资料 → 阶段成果 → 人工审批 → 最终内容包
```

## 教学设计原则

1. **一篇只引入一个主要增量。** 不为了展示功能而提前塞入后续概念。
2. **第一篇只完成纯对话闭环。** 不读取文件、不配置 Sandbox、不创建 Tool 或 Subagent。
3. **能力由业务问题推动。** 先遇到“资料无法沉淀”，再引入任务工作区；先遇到“一个岗位做不完”，再引入 Subagent。
4. **每篇结束都能运行和验收。** 文章、配套代码和测试提示必须同步更新。
5. **默认不依赖外部业务系统。** 使用随项目提供的 Markdown 和模拟素材，降低复现门槛。
6. **线上能力后置。** 本地先跑通，Vercel Sandbox、部署、观测和质量门禁集中在第 9～11 篇。
7. **不把模型输出当作事实。** 技术信息必须有资料边界，发布必须经过人工批准。
8. **每个阶段目录都是完整快照。** 从第 1 篇开始，读者直接进入 `examples/<章节>/`、安装依赖并配置模型凭据，就能运行当篇验收任务，不需要手工补做前文代码。

第 0 篇是系列导读，不承担代码运行目标；第 1～11 篇都必须满足“能启动、能操作、能产生当篇结果”三个条件。

## 系列结构（第 0～11 篇）

### 第 0 篇：认识 Eve，并公布最终项目

**标题：**《Vercel 又想复制 Next.js？这次它盯上了 AI Agent》

**任务：**建立认知，不写大段代码。

- Eve 解决的是哪一层问题；
- filesystem-first 的项目组织方式；
- 与 Vercel AI SDK、LangGraph、OpenAI Agents SDK 的能力边界；
- Public Preview 阶段的采用风险；
- 公布 SpringForAll AI 内容运营团队和最终操作界面。

**读者带走：**为什么值得继续关注这个系列。

**当前状态：**草稿已完成。

### 第 1 篇：创建第一位可对话的内容负责人

**标题：**《一个目录就是一名员工：用 Eve 创建 SpringForAll 的第一位 AI 内容负责人》

**任务：**完成最小可对话闭环。

- 使用 `--channel-web-nextjs` 初始化项目；
- 配置 Node.js、模型和环境变量；
- 编写 `agent/instructions.md`；
- 理解 `agent.ts` 与 `instructions.md` 的职责；
- 启动 Web Chat，提交一个完整的内容需求；
- 验证任务类型、读者、流程、缺失信息和当前状态；
- 验证 Agent 不会在只要求规划时直接写正文或大纲。

**明确不包含：**Sandbox、文件读取、Tools、Skills、Subagents 和部署。

**代码里程碑：**`v0.1-conversation`

**验收结果：**网页可以对话，内容负责人能按照固定结构分析任务。

**当前状态：**文章和配套代码已完成第一轮修改。

### 第 2 篇：让内容任务拥有自己的工作区

**暂定标题：**《聊天记录不等于内容资产：给 Eve Agent 建一个任务工作区》

**任务：**让 Agent 读取本地资料并保存阶段成果。

- 用“资料只存在聊天中、无法复用”引出工作区；
- 建立 `inbox → workspace → outputs` 任务目录；
- 提供 SpringForAll 账号档案和第一份内容 brief；
- 介绍 Eve Sandbox 与 `/workspace` 的关系；
- 直接选用适合 Markdown 读写的本地轻量后端，不展开后端选型；
- 让内容负责人读取资料并写出任务分析文件；
- 说明项目文件与 Sandbox 文件的边界。

**明确不包含：**自定义 Tools、Skills、Subagents、工作流恢复和线上 Sandbox。

**代码里程碑：**`v0.2-task-workspace`

**验收结果：**同一任务的输入和输出可以在项目中查看并重复执行。

**当前状态：**文章和配套代码已完成第一轮修改。

### 第 3 篇：用类型安全 Tools 管理内容任务

**暂定标题：**《别让 Agent 随便读写文件：用 Eve Tools 管理内容任务》

**任务：**把内置文件操作收敛成明确的业务动作。

- 增加创建任务、读取资料、保存阶段成果等类型安全工具；
- 约束读写目录和文件命名；
- 验证参数、错误处理与可重复执行；
- 让内容负责人通过 Tool 保存一份结构化任务卡。

**明确不包含：**Skills、内容 SOP、Subagents 和长任务恢复。

**代码里程碑：**`v0.3-task-tools`

**验收结果：**Agent 只能通过受控业务动作创建任务和保存任务卡。

### 第 4 篇：用 Skills 固化选题方法

**暂定标题：**《别把整套内容 SOP 都塞进 Prompt：用 Eve Skills 固化选题方法》

**任务：**让内容负责人按可复用的方法完成选题策划。

- 回顾 instructions、Tools 与 Skills 的职责边界；
- 编写选题策划 Skill；
- 定义选题评价维度和输出格式；
- 按需加载方法，而不是把 SOP 塞进长期指令；
- 通过上一章的 Tool 保存多个候选选题。

**明确不包含：**工作流恢复、Subagents 和人工审批。

**代码里程碑：**`v0.4-topic-skill`

**验收结果：**内容负责人能够稳定生成并保存结构化选题方案。

### 第 5 篇：让内容任务可以暂停和恢复

**暂定标题：**《等资料、等反馈、服务重启：内容 Agent 如何断点续跑》

**任务：**把一次性聊天升级为可持续的内容任务。

- 内容任务为什么会等待数小时或数天；
- Eve session 与 durable workflow；
- 等待用户补充资料并继续原任务；
- 开发服务重启后的恢复行为；
- continuation token、checkpoint 与幂等写入；
- 避免恢复后重复生成或覆盖成果。

**明确不包含：**Subagents、审批 UI 和线上部署。

**代码里程碑：**`v0.5-durable-task`

**验收结果：**任务在等待或重启后可以从原阶段继续。

### 第 6 篇：组建多 Agent 内容团队

**暂定标题：**《一个 Agent 一个目录：用 Eve 组建 AI 内容运营团队》

**任务：**把不同工作交给不同岗位。

- 创建选题策划、Java 技术研究员、技术作者和技术审校员；
- 每个 Subagent 拥有独立 instructions、模型与最小能力；
- 内容负责人进行委派、收集和汇总；
- 设计串行依赖与可并行任务；
- 处理研究与写作结论冲突；
- 控制上下文、成本和调用次数。

**明确不包含：**人工审批、运营看板和线上部署。

**代码里程碑：**`v0.6-content-team`

**验收结果：**一次内容任务可以完成策划、研究、写作和审校的多岗位流转。

### 第 7 篇：加入人工审批和退回修改

**暂定标题：**《AI 写完不等于能发布：给内容团队加上审批和退回机制》

**任务：**建立人机协作边界。

- 选题确认、大纲确认和发布批准；
- 审批请求如何暂停和恢复工作流；
- 批准、拒绝、修改后再提交；
- 审校失败后退回研究员或作者；
- 防止 Agent 把“生成完成”误判为“已经发布”；
- 保存审批记录与阶段版本。

**明确不包含：**自定义运营界面和部署。

**代码里程碑：**`v0.7-approvals`

**验收结果：**系统形成“生成—审查—修改—批准”的闭环。

### 第 8 篇：把聊天页升级成内容运营台

**暂定标题：**《不只是一块聊天框：给多 Agent 内容团队做一个运营台》

**任务：**交付可操作的业务界面。

- 创建内容任务；
- 查看各岗位状态和任务时间线；
- 浏览选题、研究底稿、草稿和审校报告；
- 在页面中批准、退回并填写修改意见；
- 预览最终 Markdown；
- 下载文章、摘要、标题备选和发布清单；
- 处理刷新、错误和任务恢复状态。

**明确不包含：**线上部署、Tracing 和 Evals。

**代码里程碑：**`v0.8-operations-console`

**验收结果：**用户无需操作目录或调用 API，即可完成一次内容生产任务。

### 第 9 篇：部署到 Vercel

**暂定标题：**《跑起来还不算完：把 Eve 内容团队部署到 Vercel》

**任务：**把本地可用系统变成线上可访问系统。

- Vercel 项目、模型凭据和环境变量；
- Preview 与 Production 部署；
- 本地后端与 Vercel Sandbox 的切换；
- 检查构建产物、资源和运行限制；
- 用 Preview 完成一次线上内容任务；
- 整理最小上线检查清单。

**明确不包含：**系统化 Tracing、Evals、CI 门禁和自动回滚。

**代码里程碑：**`v0.9-vercel-deployment`

**验收结果：**Preview 地址可以完整创建并完成一次内容任务。

### 第 10 篇：让每次 Agent 运行可观察

**暂定标题：**《Agent 为什么这么回答：用 Tracing 看懂一次内容任务》

**任务：**让运行过程、成本与失败原因可以复盘。

- 查看 Agent Runs、日志和事件流；
- 使用 OpenTelemetry 串起模型、Tool、Subagent 与审批步骤；
- 定位慢步骤、失败步骤和重复调用；
- 记录 token、延迟与阶段成本；
- 为运营台增加最小诊断入口。

**明确不包含：**自动质量判定、CI 门禁和回滚策略。

**代码里程碑：**`v0.10-observability`

**验收结果：**可以从一次任务定位每个阶段的耗时、调用与失败原因。

### 第 11 篇：用 Evals 和 CI 守住发布质量

**暂定标题：**《提示词改坏了谁知道？给 Eve 内容团队加上 Evals 和发布门禁》

**任务：**完成可评测、可发布、可回滚的生产闭环。

- 编写正常任务、资料不足、越权发布和审校失败等 Eval；
- 为关键岗位定义可检查的验收条件；
- 将 Evals 接入 CI；
- 使用 Preview 验证候选版本；
- 设计失败阻断、版本回滚和成本阈值；
- 汇总生产环境检查清单。

**代码里程碑：**`v1.0-production`

**验收结果：**得到一个线上可访问、可观察、可评测、可回滚的内容运营系统。

## 每篇统一结构

1. 上一篇留下了什么真实问题；
2. 本篇完成后能看到什么结果；
3. 只解释本篇必要的一个核心概念；
4. 修改目录和关键代码；
5. 启动并完成一次可复制测试；
6. 解释失败现象和排查方法；
7. 给出本篇验收清单；
8. 说明当前系统仍缺什么，并自然引出下一篇。

## 项目与内容约定

- 公开仓库：`https://github.com/dyc87112/vercel-eve-content-team-tutorial`；
- 系列文章统一放在 `articles/`；
- 每篇文章只展示必要代码，完整版本以对应的 `examples/<章节>/` 为准；
- 每个阶段目录都包含截至当篇的完整代码、测试素材和 README，不依赖其他阶段目录的运行产物；
- `.env.example` 只保留变量名，不提交真实密钥；
- 模型默认通过 Vercel AI Gateway 使用 `minimax/minimax-m3`；
- Eve 版本、Node.js 要求和命令在每篇发稿前重新验证；
- Git tag 和 GitHub Release 用于记录文章发布版本，不作为读者查找阶段代码的主要方式；
- 每篇 README 都提供一条最小测试输入和一组可观察的验收条件。

## 编辑风险与检查项

1. Eve 仍处于 Public Preview，发稿前必须用配套项目重新验证命令和 API。
2. 不要为了展示 Eve 功能而让单篇同时引入多个陌生概念。
3. 不把模型生成内容描述成经过事实验证的内容。
4. 不把 `vercel deploy` 等同于生产化，评测、权限、审计、成本和回滚必须覆盖。
5. 保证基础案例不依赖公众号接口、GitHub、Slack、数据库或 Java 服务。
6. 文章里的测试提示、目录结构和配套代码必须保持一致。
7. 如果系列结构再次调整，先更新本规划，再修改文章和代码。
