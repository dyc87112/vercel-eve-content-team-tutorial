# Vercel 又想复制 Next.js？这次我们用 Eve 搭一支 AI 内容团队

如果说过去两年 AI 应用开发里最热闹的是「怎么把模型接进产品」，那么接下来真正麻烦的问题会变成：

**怎么把一个 Agent 当成正常的软件项目来构建、维护和上线？**

这两个问题看起来很接近，工程量却完全不同。

调用一次模型并不复杂。写一段系统提示词，接一个模型 API，再准备几个工具，一个能聊天、能查资料、能调用接口的 Agent demo 很快就能跑起来。

但只要你准备把它放进真实工作流里，问题马上就来了：

- Agent 的角色和边界写在哪里？
- 复杂任务要不要拆成多个子 Agent？
- 选题、写作、审稿这种流程如何沉淀，而不是每次都靠临场提示词？
- 模型、网关、上下文窗口和 token 成本怎么规划？
- Agent 生成的草稿、研究笔记、执行记录应该放在哪里？
- 定时任务、评测、部署和人工审批又该怎么接进来？

这些问题没有一个能靠「再优化一下 Prompt」彻底解决。

这也是 Vercel Eve 值得关注的地方。

Eve 有点像 Vercel 想给 Agent 世界准备的一套工程框架。它不只是让你再写一个 Agent Loop，而是试图把 Agent 项目里反复出现的结构固定下来：Agent 配置、instructions、skills、subagents、tools、sandbox、schedules、evals，以及最终部署时的运行边界。

换句话说，Eve 想解决的不是：

> 怎么让模型多思考几轮？

而是：

> 怎么把一个已经能工作的 Agent，变成一个可以维护、可以拆分、可以验证、可以隔离运行产物，并且最终可以部署的系统？

这个系列就从这个问题开始。

## Agent Demo 很多，真正能长期维护的不多

现在做一个 Agent demo 已经不难了。

给它一段角色说明，让它调用模型，必要时搜索资料或读写文件，再把结果返回给用户。这样的 demo 很容易让人兴奋，因为它看起来已经具备了「自主工作」的雏形。

但真实项目不是一次演示。

真实项目会遇到更琐碎也更要命的问题：

- 今天改了一句系统指令，明天怎么知道旧能力有没有退化？
- 一个任务需要研究、写作、审稿三个阶段，应该用一个长上下文硬扛，还是拆成不同角色？
- Agent 需要生成文件时，应该写进项目目录，还是写进隔离工作区？
- 选题建议到底来自模型记忆，还是来自最新官方资料和可追溯来源？
- 模型上下文窗口不够时，哪些内容应该保留，哪些应该丢弃？
- 接入自有模型网关后，如何确认 base URL、模型 ID、token 预算都是正确的？

这些问题才是 Agent 从 demo 走向工程项目时真正会撞到的墙。

所以这个系列不会从「写一个最聪明的 Prompt」开始。

我们会从一个更工程化的目标开始：用 Eve 搭一支可以逐步演进的 AI 内容团队。

## 为什么是 SpringForAll 内容团队

这个系列选择的案例是 SpringForAll 内容团队 Agent。

SpringForAll 面向中文 Java / Spring 开发者。这个读者群体很适合拿来做 Agent 案例，因为他们对内容质量的要求很具体：

- 选题不能只是 AI 热点，要和 Java / Spring 开发有关；
- 文章不能只有概念，要能落到工程实践；
- 结论不能凭空生成，要有官方文档或一手资料支撑；
- 依赖版本、API 名称、配置项、迁移风险都要说清楚；
- 最终发布前必须有人审稿，不能让 Agent 自己宣布已经完成发布。

换句话说，这不是一个「让 AI 写一篇公众号」的问题。

它更像一个内容团队工作流：

- 先研究最近有哪些值得关注的技术信号；
- 再判断哪些选题适合 SpringForAll 的读者；
- 然后把选题扩展成文章结构；
- 再生成草稿；
- 最后检查来源、事实、版本、可运行性和发布风险。

这正好可以把 Eve 的核心概念自然引出来。

主 Agent 可以像内容负责人一样负责目标和调度；skills 可以沉淀选题、写作、审稿的方法；subagents 可以拆出 researcher、writer、reviewer；sandbox 可以隔离研究笔记和草稿；后续 tools、schedules、evals、部署也都能围绕同一个案例继续扩展。

这比单纯做一个聊天机器人更贴近真实 Agent 项目。

## Eve 最值得关注的不是「又一个调用模型的方法」

如果只是调用模型，我们已经有很多选择。

你可以直接用模型厂商 SDK，也可以用 Vercel AI SDK，还可以接自己的 OpenAI-Compatible 网关。单次调用、流式输出、结构化结果、Tool Calling，这些能力已经越来越成熟。

Eve 的价值不在这里。

它更像站在应用外层，回答这些问题：

- Agent 项目应该如何组织目录？
- 哪些指令始终生效，哪些流程按需加载？
- 多个子 Agent 如何表达成可维护的文件结构？
- 运行过程中的文件和命令应该在哪里执行？
- 定时任务和评测应该如何成为项目的一部分？
- 未来部署到 Vercel 时，Agent 如何继续保持同样的结构？

这也是我觉得 Eve 有意思的地方：它把 Agent 当作一个软件项目来管理，而不是把 Agent 藏在某个控制台里的一堆配置项中。

当 instructions、skills、subagents、schedules、evals 都变成普通文件之后，它们就可以进入 Git，可以被 review，可以看 diff，也可以随着教程一步步演进。

这对长期维护非常重要。

因为 Agent 的行为不是只由代码决定，也由指令、流程、工具和上下文边界共同决定。如果这些东西不进入工程体系，Agent 就很容易变成一个「能跑，但没人敢改」的黑盒。

## 这个系列最终会得到什么

这个系列前半段会先完成一个阶段性 demo：SpringForAll 内容团队 Agent。

它不是最终的生产系统，但会具备一个业务型 Agent 项目该有的基本骨架：

1. 一个可以在本地 CLI 中对话的基础 Agent；
2. 一套基于 Vercel AI Gateway 的默认模型方案；
3. 一套可切换到自定义 AI Provider 的配置；
4. 一组围绕内容运营设计的 skills；
5. researcher、writer、reviewer 三个子 Agent；
6. 一个明确的 sandbox 运行边界；
7. 一个不会自动发布、保留人工审批的内容工作流。

注意最后一点很重要。

这个 Agent 不是自动发文机器。

技术内容里，自动发布尤其危险。版本号、依赖坐标、API 名称、配置项、性能结论、安全结论，都需要人工核验。Agent 可以帮助内容团队更快完成资料收集、结构整理、草稿准备和审稿清单，但最终编辑责任仍然在人。

所以我们要搭建的是一支「辅助内容团队」，不是一个「无人值守发布系统」。

## 先完成 00 到 04 篇，再继续扩展

这个系列当前会先写 `00` 到 `04` 这 5 篇，对应 demo 已经实现到的部分。

第一篇，也就是这篇前言，先介绍 Eve，并解释为什么选择 SpringForAll 内容团队作为案例。

第二篇会创建第一个 Agent。我们会走完整的 Vercel 方案，使用 Vercel AI Gateway 配置模型，让 Agent 能在本地 CLI 中完成基础 chat。这个阶段只关注最小可运行结构，不引入复杂能力。

第三篇会接入自定义 AI Provider。真实项目不会永远只用一个默认模型或默认网关，所以我们会支持自己的 OpenAI-Compatible gateway，同时讨论 coding plan、token plan、上下文窗口和成本边界。

第四篇会使用 subagents 和 skills 构建 AI 内容团队。我们会把选题规划、文章写作、审稿检查拆成 skills，再把 researcher、writer、reviewer 拆成不同子 Agent。到这里，Agent 才真正开始像一个团队。

第五篇会使用 sandbox 隔离运行产物。Agent 在研究和写作过程中会产生资料、草稿、审校记录和临时文件，这些不应该混进 Agent 定义代码。我们会让运行产物进入 sandbox 的 `/workspace`，让仓库只管理教程、样例和 Agent 本身。

`00` 到 `04` 篇完成后，读者会看到一个完整的阶段性结果。

后续先保留 4 个方向，不在前半段展开：

- Tools：自定义更丰富的工具来支撑内容团队目标；
- Schedules：每天自动生成内容雷达；
- Evals：给 Agent 工作流加最低限度质量门槛；
- 部署到 Vercel：把本地 Agent 变成可运行服务。

这样安排的好处是节奏更稳。

我们不会一开始就把 tools、定时任务、评测、部署全部塞进去，而是先把 Agent 的身份、模型、团队结构和运行边界讲清楚。否则读者很容易看到一堆目录和概念，却不知道每一层为什么存在。

## 仓库会如何组织

这个教程仓库会分成三个目录。

`tutorial/` 放系列文章。

`example/` 放每篇文章对应的最小样例工程。

`demo/` 放阶段性完整成品。当前的完整尝试会放在 `demo/final-content-agent/`，作为 `00` 到 `04` 篇完成后的参考目标。

这个结构本身也是教程的一部分。

Agent 项目很容易变乱。文章、示例工程、运行缓存、模型输出、研究草稿、最终 demo 如果全部混在一起，几轮迭代之后，就很难判断一个文件到底应该被版本管理，还是只是某次运行的产物。

从第一天开始把教程、样例和 demo 分开，后面讲 sandbox 和 git 边界时就顺理成章。

## 适合谁读

这个系列适合三类读者。

第一类是想了解 Eve 的开发者。你可能已经写过模型调用或 Agent demo，但还想知道 Eve 这种框架如何组织一个更完整的 Agent 项目。

第二类是想把 Agent 放进真实业务流程的后端开发者。你关心的不只是模型回答得好不好，还包括配置、成本、上下文、任务拆分、运行隔离和部署边界。

第三类是做技术内容、开发者关系或社区运营的人。你可能并不想让 AI 自动发布文章，但希望它能帮你更系统地完成选题、资料整理、草稿和审稿准备。

如果你的目标只是快速写一个聊天 demo，这个系列可能会显得有点慢。

但如果你想看一个 Agent 项目如何从最小可运行结构一步步长成可维护系统，这个系列会更合适。
